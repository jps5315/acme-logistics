# Requirements Document

## Introduction

This feature replaces the static single-file HTML/JS dashboard for Acme Logistics Carrier Sales with a dynamic React-based frontend and an enhanced FastAPI backend. The upgrade introduces financial KPIs (Gross Profit and Gross Profit Margin), enriched call data (classifications, sentiment, per-call summaries), and AI-generated insights powered by Google Gemini. The HappyRobot workflow continues to POST call results to the backend, which now stores richer data, aggregates metrics, queries Gemini for insights, and serves everything to the React frontend via a REST API.

---

## Glossary

- **Dashboard**: The React single-page application that displays carrier sales metrics and AI insights.
- **Backend**: The Python FastAPI application (`main.py`) that receives call data, aggregates metrics, and serves the Dashboard.
- **HappyRobot**: The external voice-agent workflow that POSTs call result data to the Backend after each carrier call.
- **Call_Result**: A structured data payload sent by HappyRobot to the Backend after a carrier call completes.
- **Call_Summary**: A human-readable text summary of an individual carrier call, included in the Call_Result.
- **KPI**: Key Performance Indicator — a quantitative metric displayed on the Dashboard.
- **Gross_Profit**: The financial gain per call, calculated as the difference between the agreed carrier rate and the loadboard reference rate.
- **Gross_Profit_Margin**: Gross Profit expressed as a percentage of the agreed carrier rate.
- **Call_Classification**: A categorical label assigned to a call outcome (e.g., "successful", "failed", "could do better").
- **Sentiment**: A categorical label describing the carrier's emotional tone during a call (e.g., "happy", "unsatisfied", "interested").
- **Gemini_Client**: The Backend component responsible for calling the Google Gemini API.
- **AI_Insights**: A text block generated by the Gemini_Client and displayed on the Dashboard, summarising trends and recommendations based on current KPIs.
- **Metrics_Endpoint**: The Backend's `GET /metrics` endpoint that returns aggregated KPIs, charts data, recent calls, and AI_Insights.
- **Calls_Endpoint**: The Backend's `POST /calls/result` endpoint that receives Call_Result payloads from HappyRobot.

## Requirements

### Requirement 1: Enriched Call Result Ingestion

**User Story:** As a HappyRobot workflow operator, I want to POST enriched call data (including gross profit figures, call classification, sentiment, and a call summary) to the backend, so that the dashboard can display complete financial and qualitative information for every call.

#### Acceptance Criteria

1. WHEN a POST request is received at `/calls/result`, THE Calls_Endpoint SHALL accept a payload containing: `mc_number`, `carrier_name`, `load_id`, `agreed_price`, `loadboard_rate`, `deal_outcome`, `customer_sentiment`, `call_summary`, `gross_profit`, `gross_profit_margin`, `call_duration`, `session_id`, `transcript`, `notes`, and `timestamp`.
2. WHEN a POST request payload omits optional fields, THE Calls_Endpoint SHALL store the record with those fields set to null without returning an error.
3. WHEN a valid Call_Result is received, THE Calls_Endpoint SHALL persist the record to durable storage and return a JSON response containing `success: true` and the assigned record `id`.
4. IF a POST request to `/calls/result` is received without a valid API key, THEN THE Calls_Endpoint SHALL return HTTP 403.
5. IF a POST request body cannot be parsed as a valid Call_Result, THEN THE Calls_Endpoint SHALL return HTTP 422 with a descriptive validation error.

---

### Requirement 2: Financial KPI Aggregation

**User Story:** As a sales manager, I want the backend to compute and expose Gross Profit and Gross Profit Margin KPIs, so that I can monitor the financial performance of carrier negotiations at a glance.

#### Acceptance Criteria

1. WHEN `GET /metrics` is called, THE Metrics_Endpoint SHALL return `total_gross_profit` as the sum of all stored `gross_profit` values across all Call_Results.
2. WHEN `GET /metrics` is called, THE Metrics_Endpoint SHALL return `avg_gross_profit_margin` as the mean of all stored `gross_profit_margin` values, rounded to one decimal place.
3. WHEN no Call_Results have been stored, THE Metrics_Endpoint SHALL return `total_gross_profit` as `0` and `avg_gross_profit_margin` as `0.0`.
4. WHEN `GET /metrics` is called without a valid API key, THE Metrics_Endpoint SHALL return HTTP 403.

---

### Requirement 3: Per-Call Summary Exposure

**User Story:** As a sales manager, I want to see a human-readable summary for each individual call in the dashboard, so that I can quickly understand what happened on each call without reading a full transcript.

#### Acceptance Criteria

1. WHEN `GET /metrics` is called, THE Metrics_Endpoint SHALL include a `recent_calls` array in the response where each entry contains the `call_summary` field alongside existing call fields.
2. WHEN a Call_Result was stored without a `call_summary`, THE Metrics_Endpoint SHALL return `null` for that entry's `call_summary` field.

---

### Requirement 4: Gemini AI Insights Generation

**User Story:** As a sales manager, I want the backend to automatically generate AI-powered insights from current KPIs using Google Gemini, so that I receive actionable recommendations without manually analysing the data.

#### Acceptance Criteria

1. WHEN `GET /metrics` is called and at least one Call_Result exists, THE Gemini_Client SHALL send a prompt to the Google Gemini API containing: `total_calls`, `success_rate_pct`, `total_gross_profit`, `avg_gross_profit_margin`, `outcomes` breakdown, and `sentiments` breakdown.
2. WHEN the Gemini API returns a successful response, THE Metrics_Endpoint SHALL include the response text as `ai_insights` in the `GET /metrics` response.
3. IF the Gemini API call fails or times out, THEN THE Metrics_Endpoint SHALL return `ai_insights` as `null` and SHALL NOT return an error to the caller.
4. WHEN no Call_Results have been stored, THE Metrics_Endpoint SHALL return `ai_insights` as `null` without calling the Gemini API.
5. WHERE a `GEMINI_API_KEY` environment variable is not configured, THE Backend SHALL log a warning at startup and THE Metrics_Endpoint SHALL return `ai_insights` as `null`.

---

### Requirement 5: React Dashboard — Financial KPI Display

**User Story:** As a sales manager, I want the React dashboard to display Gross Profit and Gross Profit Margin KPI cards, so that I can monitor financial performance alongside existing operational KPIs.

#### Acceptance Criteria

1. THE Dashboard SHALL display a KPI card labelled "Gross Profit" showing `total_gross_profit` formatted as a dollar amount.
2. THE Dashboard SHALL display a KPI card labelled "Gross Profit Margin" showing `avg_gross_profit_margin` formatted as a percentage.
3. WHEN the `GET /metrics` response is received, THE Dashboard SHALL update all KPI card values without requiring a full page reload.
4. WHILE data is being fetched from the Metrics_Endpoint, THE Dashboard SHALL display a loading indicator in place of KPI values.
5. IF the Metrics_Endpoint returns an error, THEN THE Dashboard SHALL display an error message and retain the last successfully loaded values.

---

### Requirement 6: React Dashboard — Call Classification and Sentiment Charts

**User Story:** As a sales manager, I want the React dashboard to display call classification and sentiment breakdown charts, so that I can visually assess deal outcomes and carrier mood trends.

#### Acceptance Criteria

1. THE Dashboard SHALL render a chart displaying the count of calls per Call_Classification category using data from the `outcomes` field of the Metrics_Endpoint response.
2. THE Dashboard SHALL render a chart displaying the count of calls per Sentiment category using data from the `sentiments` field of the Metrics_Endpoint response.
3. WHEN the Metrics_Endpoint returns updated data, THE Dashboard SHALL re-render both charts to reflect the latest values.
4. WHEN a chart category has a count of zero, THE Dashboard SHALL still display that category in the chart with a value of zero.

---

### Requirement 7: React Dashboard — Per-Call Summary Table

**User Story:** As a sales manager, I want the React dashboard to display a table of recent calls that includes each call's summary, so that I can review individual call outcomes and context in one place.

#### Acceptance Criteria

1. THE Dashboard SHALL display a table of recent calls containing: timestamp, MC number, carrier name, load ID, loadboard rate, agreed price, deal outcome, sentiment, and call summary.
2. WHEN a call's `call_summary` is null, THE Dashboard SHALL display "—" in the call summary column.
3. WHEN the recent calls list is empty, THE Dashboard SHALL display an empty-state message in place of the table.
4. THE Dashboard SHALL display the most recent calls first, ordered by timestamp descending.

---

### Requirement 8: React Dashboard — AI Insights Panel

**User Story:** As a sales manager, I want the React dashboard to display AI-generated insights from Gemini, so that I can act on data-driven recommendations without leaving the dashboard.

#### Acceptance Criteria

1. THE Dashboard SHALL display an AI Insights panel that renders the `ai_insights` text returned by the Metrics_Endpoint.
2. WHEN `ai_insights` is null, THE Dashboard SHALL display a placeholder message indicating that insights are unavailable.
3. WHEN the Metrics_Endpoint returns a new `ai_insights` value, THE Dashboard SHALL update the AI Insights panel without requiring a full page reload.
4. WHILE the Metrics_Endpoint request is in flight, THE Dashboard SHALL display a loading state in the AI Insights panel.

---

### Requirement 9: React Dashboard — Auto-Refresh

**User Story:** As a sales manager, I want the dashboard to automatically refresh its data at a regular interval, so that I always see up-to-date metrics without manually triggering a reload.

#### Acceptance Criteria

1. THE Dashboard SHALL automatically poll the Metrics_Endpoint at a configurable interval, with a default of 30 seconds.
2. WHEN a polling request completes successfully, THE Dashboard SHALL update all displayed data and show the time of the last successful update.
3. IF a polling request fails, THEN THE Dashboard SHALL display an error indicator and continue polling at the configured interval.

---

### Requirement 10: Backend Configuration and Deployment

**User Story:** As a DevOps engineer, I want the backend and React frontend to be configurable via environment variables and deployable via Docker Compose, so that the system can be promoted across environments without code changes.

#### Acceptance Criteria

1. THE Backend SHALL read the Google Gemini API key from a `GEMINI_API_KEY` environment variable.
2. THE Backend SHALL read the API authentication key from an `API_KEY` environment variable, defaulting to `acme-secret-key-2024` when not set.
3. THE Dashboard SHALL read the backend base URL from a build-time or runtime environment variable, defaulting to `http://localhost:8000`.
4. THE docker-compose.yml SHALL define services for the Backend and the React Dashboard such that `docker compose up --build` starts the full system.
5. WHERE the `GEMINI_API_KEY` environment variable is not set, THE Backend SHALL start successfully and serve all endpoints, returning `ai_insights: null` from the Metrics_Endpoint.
